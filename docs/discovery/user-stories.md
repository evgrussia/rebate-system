## Модуль 1: Регистрация и аутентификация

### US-001: Регистрация через email (Web)

**Приоритет:** P0 | **Persona:** Newcomer Maria | **FR:** FR-001

**Story:**
As a новый пользователь, I want зарегистрироваться через email и пароль, so that я могу получить доступ к rebate-сервису.

#### Context
Пользователь впервые попал на сайт CryptoRebate и хочет начать пользоваться сервисом.

#### Preconditions
- Пользователь не имеет аккаунта в системе
- Пользователь на странице /register

#### Acceptance Criteria

**Scenario 1: Happy Path**
```
Given: Пользователь на странице регистрации
When: Вводит валидный email и пароль (мин. 8 символов, 1 цифра, 1 буква) и нажимает "Зарегистрироваться"
Then: Аккаунт создан, пользователь автоматически авторизован и перенаправлен на /dashboard
```

**Scenario 2: Email уже занят**
```
Given: Пользователь на странице регистрации
When: Вводит email, который уже зарегистрирован в системе
Then: Отображается ошибка "Этот email уже зарегистрирован", предлагается войти
```

**Scenario 3: Невалидный пароль**
```
Given: Пользователь на странице регистрации
When: Вводит пароль короче 8 символов или без цифры
Then: Отображается ошибка с требованиями к паролю, кнопка "Зарегистрироваться" неактивна
```

#### Dependencies
- Нет

## Модуль 3: Exchange Integration

### US-009: Синхронизация данных с биржи

**Приоритет:** P0 | **Persona:** System | **FR:** FR-020–FR-025

**Story:**
As a система, I want периодически синхронизировать данные с бирж, so that расчёт rebate основан на актуальных данных.

#### Acceptance Criteria

**Scenario 1: Happy Path**
```
Given: Настроена интеграция с биржей MEXC, есть привлечённые трейдеры
When: Наступает время синхронизации (по расписанию)
Then: Система запрашивает API биржи, получает данные о комиссиях и объёмах, сохраняет в БД
```

**Scenario 2: API недоступен**
```
Given: API биржи возвращает ошибку или timeout
When: Система выполняет запрос
Then: Ошибка логируется, запланирована повторная попытка через N минут (retry), админ уведомлён
```

**Scenario 3: Данные расходятся**
```
Given: Полученные данные отличаются от предыдущей синхронизации более чем на 20%
When: Система обрабатывает ответ
Then: Данные сохраняются с флагом "requires_review", админ уведомлён
```

## Модуль 4: Rebate Calculator

### US-012: Автоматический расчёт rebate

**Приоритет:** P0 | **Persona:** System | **FR:** FR-030, FR-033

**Story:**
As a система, I want автоматически рассчитывать rebate для каждого трейдера, so that трейдеры видят актуальный баланс.

#### Acceptance Criteria

**Scenario 1: Happy Path**
```
Given: Данные о комиссиях трейдера за период синхронизированы с биржи
When: Запускается расчёт rebate
Then: Для каждого трейдера рассчитана его доля: rebate = комиссия_биржи × ставка_биржи × доля_трейдера, результат записан в историю начислений, баланс обновлён
```

**Scenario 2: Нулевые данные**
```
Given: Трейдер не совершал сделок в периоде
When: Запускается расчёт
Then: Начисление = 0, запись создана для аудита
```

## Модуль 7: Payout Engine

### US-026: Создание заявки на вывод

**Приоритет:** P0 | **Persona:** Trader Alex | **FR:** FR-060, FR-061

**Story:**
As a трейдер, I want создать заявку на вывод с указанием суммы, адреса и сети, so that я могу получить rebate на свой кошелёк.

#### Acceptance Criteria

**Scenario 1: Happy Path**
```
Given: Трейдер имеет баланс 50 USDT, минимум вывода = 10 USDT
When: Создаёт заявку на 30 USDT, адрес TRC-20
Then: Заявка создана, статус "pending", баланс заморожен на 30 USDT
```

**Scenario 2: Ниже минимума**
```
Given: Минимум = 10 USDT, трейдер вводит 5 USDT
When: Отправляет заявку
```
